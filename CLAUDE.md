## Architecture

- `trello_cli/config.py` — credentials + local-backend root, stored in `~/.trello-cli.json`. Board and backend *selection* are per-invocation only (flags/env), never persisted (see Statelessness)
- `trello_cli/api.py` — thin **facade**; each operation forwards to the active backend via `get_backend()`. `main.py`'s `api.<op>(...)` call sites stay unchanged
- `trello_cli/backends/` — pluggable data sources behind a common interface:
  - `base.py` — the `Backend` ABC: the ~40 operations the CLI needs, all returning Trello-shaped dicts
  - `trello.py` — `TrelloBackend`, the httpx client over the Trello REST API (requests only the fields each command needs)
  - `local.py` — `LocalBackend`, a self-hosted file store. Backs **every** CLI command: boards/lists/cards (CRUD + move/pos/archive/rename/desc/due), labels, checklists, comments, attachments, members, `card mine`, and `activity`/`updates` read from the log. Labels are stored on cards as `idLabels` and resolved to full dicts from `labels.json` at read time (so `label edit`/`delete` reflect everywhere); comments/checklists/attachments live inline in the card JSON; members are a single OS-username user, so `mine` returns every open card. **Persisted per-list `sort`** (the web auto-sort feature, `LIST_SORTS` = `manual`/`newest`/`oldest`/`name`, default `manual`): stored as an optional `sort` key on each `lists.json` entry — written by `create_list`/`create_board`/`import_board`, `setdefault`-ed in `get_lists` so pre-`sort` stores never KeyError, validated + persisted in `update_list(sort=…)` (which also one-shot re-sorts the list's existing cards), and applied in `create_card` (`_auto_place_pos` overrides the requested `pos` so a new card lands in its sorted slot). Trello has no native per-list sort, so it's **local-only** — the Trello backend drops a `sort` field from `update_list` (clean no-op). Also `import_board(...)` — local-only (not on the ABC), the target of `trello export`: writes a source backend's Trello-shaped board/lists/labels/cards into the store, preserving ids and pruning cards deleted upstream (plus their attachment blob dirs). Two more local-only methods back maintenance: `gc(...)` (the `local gc` sweep: orphaned attachment dirs/files, optional `activity.log` trim) and `delete_board(...)` (the `local rm` board-folder delete), both report-then-delete so the command can offer a dry run. **Every mutator runs under the store lock** — the `_MUTATORS` tuple at the bottom of the module is the authoritative list of writers (including `gc`/`delete_board`), each wrapped once via `_locked`; reads stay lock-free
  - `store.py` — file-store primitives for `LocalBackend`: 24-hex id gen, atomic JSON/text writes (temp + `os.replace`), float `pos` midpoint math (step 65536), the `needs_rebalance`/`even_positions` collapse-guard (float midpoints can only be halved ~50× before two positions become the same float — `LocalBackend` respreads a list to even spacing once any gap drops below `MIN_GAP`, on every card/list create + pos-update; when a respread fires on a `pos`/move update, `update_card`/`update_list` tag the **returned** dict with a transient `rebalanced: true` — set after the atomic save so it is never persisted, consumed only by the web client to trigger a full reload of the now-stale siblings, ignored by the CLI), append-only `activity.log` (JSONL; written, read, and `tail_activity`-trimmed for retention), the `StoreLock` (cross-process advisory lock, see Concurrency below), and the on-disk layout (`<root>/<boardId>/{board.json, lists.json, labels.json, cards/<id>.json, attachments/<cardId>/…, activity.log}` + a top-level `<root>/.lock`)
  - `__init__.py` — `get_backend()` factory (cached singleton); selects `trello` (default) or `local` from `--backend` / `TRELLO_BACKEND` (see DESIGN.md)
- `trello_cli/fmt.py` — compact table/detail formatting and small helpers (`short_id`, `truncate`, `due_str`, `label_str`, `is_image`, `size_str`, `print_json`); backend-agnostic (formats plain dicts)
- `trello_cli/main.py` — CLI entry point, noun-group dispatch (`card`, `list`, `label`, `checklist`, `comment`, `attachment`), name/ID prefix resolution. The `export` command bridges two backends in one invocation — it reads a board snapshot via the `api` facade (shared `_gather_board` helper) and writes via a directly-instantiated *other* backend, bypassing the `get_backend()` singleton — and runs in two directions on `--to`:
  - `--to local` (default, `_export_to_local`): pull from the selected backend (must not be local) into a `LocalBackend`. Also downloads uploaded attachment blobs (default on; `--no-attachments` skips) via the *source* backend's authed `download_attachment`, rewriting each card's attachment url root-relative *before* `import_board` persists it — best-effort (per-blob failure warns + keeps the remote url; immutable blobs are reused on re-export). IDs preserved → idempotent in-place refresh
  - `--to trello` (`_export_to_trello` → `_push_board_to_trello`): push the local store (source backend must be `local`) **up to Trello as a brand-new board**. Trello mints fresh ids, so ids can't be preserved — it builds old→new maps for labels/lists, then re-creates each card and its children (comments, checklists+items, attachments) under the new ids via the ABC's write ops. **Create-new-each-time** (non-idempotent; no local→trello map persisted — keeps statelessness). Re-uploads local blobs (`add_attachment_file`), re-links URL attachments; comments are re-posted as the token user with an `_originally <author>, <date>_` provenance prefix (Trello can't set author/date); `idMembers` dropped; only open lists pushed (`--name` overrides the new board name). A tracked-mapping re-sync (`--into <board>`) is intentionally out of scope (DESIGN.md)
- `trello_cli/web/` — optional web app (the `[web]` extra: fastapi, uvicorn, watchdog), launched by `trello serve`:
  - `server.py` — FastAPI JSON API mapping 1:1 onto the `api` facade, so it renders **either** backend with no per-backend code. Endpoints: `GET /api/boards`, `GET /api/boards/{id}` (board+lists+cards), `POST /api/boards/{id}/lists` (add column), `GET /api/cards/{id}` (detail+comments), `PATCH /api/cards/{id}` (drag move/reorder), `PATCH /api/lists/{id}` (column reorder `pos`, archive `closed`, persisted `sort`), `POST /api/lists/{id}/cards` (create), `GET /api/events` (SSE live-refresh stream). Backend `SystemExit` (not-found) is translated to HTTP 404; mutating fields are whitelisted (`_CARD_PATCH_FIELDS` = `{idList, pos}`; `_LIST_PATCH_FIELDS` = `{pos, closed, sort}`). **Auth:** loopback binds are open; a non-loopback `serve --host` is token-gated — `serve(token=…)` (auto-minted if absent) installs middleware that requires the token (`Authorization: Bearer` **or** `?token=` — the query param is the only channel a browser nav / `EventSource` has) on `/api/*` only, leaving the static shell public so `app.js` can load and then attach the token itself
  - `live.py` — live-refresh plumbing: a `watchdog` observer on the local store root bumps a monotonic version counter; the `GET /api/events` async generator polls it and emits `event: change` (a counter polled across the thread boundary, no event-loop hand-off). Local-backend only — Trello has no local files, so its stream is keep-alive only
  - `static/` — vanilla JS + SortableJS (vendored under `static/vendor/`, no build step): columns + cards, drag-drop reorder/move computing the float `pos` midpoint client-side, a read-only card detail panel, and an `EventSource` on `/api/events` that reloads the board on a `change` event (skipped mid-drag). A drop also reloads the board immediately when its move/reorder PATCH response carries `rebalanced: true` (see the collapse-guard note under `store.py`/`local.py`) — without it only the dragged element's `data-pos` updates and the respread siblings stay stale until the SSE reload. **Column management:** an "Add another list" affordance after the last column (`addListEl`, `POST /api/boards/{id}/lists`), a per-column `⋯` actions menu with **Delete list** (archive via `PATCH {closed:true}`), and a per-column **sort `<select>`** (Manual/Newest/Oldest/Name) that persists via `PATCH {sort}`. A manual card drag onto an auto-sorted column **clears that column's saved sort** back to `manual` (so the user's hand-placement isn't immediately overridden) — done web-side in the card `onEnd`

## Conventions

- **Noun-group dispatch** — `_dispatch(group, subcmds, args)` routes verbs within a group. Bare nouns (or nouns followed by a non-verb) fall back to `ls` if the group has one, so `trello list` ≡ `trello list ls`.
- **Resolvers** — every domain has a `_resolve_*` helper that accepts an ID, an ID prefix, or a case-insensitive name prefix, and raises `SystemExit` on miss/ambiguity.
- **Board scope** — `_require_board()` resolves the board from `--board <name_or_id>` (parsed in `main()`) or the `TRELLO_BOARD` env var, and errors if neither is set. There is no stored "active board" (see Statelessness).
- **Backend scope** — `get_backend_name()` selects the data source from `--backend <trello|local>` (parsed in `main()`) or the `TRELLO_BACKEND` env var, defaulting to `trello`. The local store folder resolves as `--local-root` flag > `TRELLO_LOCAL_ROOT` env > config `local_root` > `~/Dropbox/trello-cli`; `trello local init [path]` creates it and persists `local_root`.
- **Statelessness (design guideline)** — the CLI is used by many agents and projects concurrently, so it keeps **no shared mutable session state**. Selection (board, backend) is always per-invocation via flags/env; only stable config (credentials, `local_root`) is persisted. Don't add "active X" state — that creates cross-invocation conflicts. (The legacy active-board feature was removed for exactly this reason; everything uses `--board`.)
- **Output mode** — `--json` is stripped in `main()` and toggles `_JSON_MODE`; read commands branch on `_is_json()` to emit raw JSON via `print_json` instead of formatted tables.
- **Backend seam** — commands never touch a concrete backend or transport. They call `api.<op>(...)`, which forwards to `get_backend()`. Every backend returns the same Trello-shaped dicts (the keys `fmt.py` reads), so adding a backend means implementing the `Backend` ABC — no command or formatter changes.
- **Concurrency (local backend)** — many agents run the CLI at once, so every `LocalBackend` *write* is a read-modify-write that would otherwise lose updates (and crash on Windows' `os.replace`) when two processes race the same file. `StoreLock` (in `store.py`) serializes them: a cross-process OS advisory lock on `<root>/.lock` (`fcntl.flock` / `msvcrt.locking`, auto-released on process death) plus an in-process `threading.RLock` for re-entrancy + thread-safety, with a bounded blocking wait. It wraps the whole load→modify→save of each mutator (the `_MUTATORS` list). **Adding a new mutating op? Add its method name to `_MUTATORS`.** Reads are intentionally lock-free (atomic writes already give each file a consistent view). Scope: same-machine multi-process. Cross-machine Dropbox edits stay last-write-wins (OS locks don't cross machines) — see DESIGN.md.
