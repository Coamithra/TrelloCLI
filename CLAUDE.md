## Architecture

- `trello_cli/config.py` â€” credentials + local-backend root, stored in `~/.trello-cli.json`. Board and backend *selection* are per-invocation only (flags/env), never persisted (see Statelessness)
- `trello_cli/api.py` â€” thin **facade**; each operation forwards to the active backend via `get_backend()`. `main.py`'s `api.<op>(...)` call sites stay unchanged
- `trello_cli/backends/` â€” pluggable data sources behind a common interface:
  - `base.py` â€” the `Backend` ABC: the ~40 operations the CLI needs, all returning Trello-shaped dicts
  - `trello.py` â€” `TrelloBackend`, the httpx client over the Trello REST API (requests only the fields each command needs). Trello has no atomic move-and-return, so `grab_top_card` **fakes** the claim handshake (CONTRIBUTING.md style): grab the top card, post a `I am doing this now â€” claim <id>` comment, wait a randomized 10-30s, then let the *earliest* in-window claim win â€” rollback (retract the comment + return the card to the source list) happens only for failures/interrupts *before* adjudication; after it, a loss just retracts its own comment and retries the next card (the winner owns the card) and a win is never undone (`_won_claim`/`_parse_dt` + the `_GRAB_*` tunables; the algorithm mirrors CONTRIBUTING.md's known-good handshake but is **not** verified against live Trello)
  - `local.py` â€” `LocalBackend`, a self-hosted file store. Backs **every** CLI command: boards/lists/cards (CRUD + move/pos/archive/rename/desc/due), labels, checklists, comments, attachments, members, `card mine`, and `activity`/`updates` read from the log. **`update_board`** (a mutator on the ABC, backing `board rename`/`board archive`/`board restore` + the web `PATCH /api/boards/{id}`) renames and/or sets the board's `closed` flag â€” archiving is a **soft delete**: the board drops out of `get_boards()` (which now takes `include_closed=False`; pass `True` for the archived/recycling-bin view) but its files stay on disk, so `restore` (`closed:false`) brings it back intact; the irreversible folder wipe is still `delete_board`. Labels are stored on cards as `idLabels` and resolved to full dicts from `labels.json` at read time (so `label edit`/`delete` reflect everywhere); comments/checklists/attachments live inline in the card JSON; members are a single OS-username user, so `mine` returns every open card. **Atomic `grab_top_card`** (backing `trello grab`): a mutator that, under the store lock, pops the top open card of a source list and moves it to a dest list in one read-modify-write â€” so concurrent grabbers each get a *distinct* card with no claim dance (the local native answer to the race CONTRIBUTING.md's handshake works around; same-machine only, Dropbox cross-machine is out of scope). **Persisted per-list `sort`** (the web auto-sort feature, `LIST_SORTS` = `manual`/`newest`/`oldest`/`name`, default `manual`): stored as an optional `sort` key on each `lists.json` entry â€” written by `create_list`/`create_board`/`import_board`, `setdefault`-ed in `get_lists` so pre-`sort` stores never KeyError, validated + persisted in `update_list(sort=â€¦)` (which also one-shot re-sorts the list's existing cards), and applied in `create_card` (`_auto_place_pos` overrides the requested `pos` so a new card lands in its sorted slot). Trello has no native per-list sort, so it's **local-only** â€” the Trello backend drops a `sort` field from `update_list` (clean no-op). Also `import_board(...)` â€” local-only (not on the ABC), the target of `trello export`: writes a source backend's Trello-shaped board/lists/labels/cards into the store, preserving ids and pruning cards deleted upstream (plus their attachment blob dirs). Two more local-only methods back maintenance: `gc(...)` (the `local gc` sweep: orphaned attachment dirs/files, optional `activity.log` trim) and `delete_board(...)` (the `local rm` board-folder delete), both report-then-delete so the command can offer a dry run. **Every mutator runs under the store lock** â€” the `_MUTATORS` tuple at the bottom of the module is the authoritative list of writers (including `update_board`/`grab_top_card`/`gc`/`delete_board`), each wrapped once via `_locked`; reads stay lock-free
  - `store.py` â€” file-store primitives for `LocalBackend`: 24-hex id gen, atomic JSON/text writes (temp + `os.replace`), float `pos` midpoint math (step 65536), the `needs_rebalance`/`even_positions` collapse-guard (float midpoints can only be halved ~50Ã— before two positions become the same float â€” `LocalBackend` respreads a list to even spacing once any gap drops below `MIN_GAP`, on every card/list create + pos-update; when a respread fires on a `pos`/move update, `update_card`/`update_list` tag the **returned** dict with a transient `rebalanced: true` â€” set after the atomic save so it is never persisted, consumed only by the web client to trigger a full reload of the now-stale siblings, ignored by the CLI), append-only `activity.log` (JSONL; written, read, and `tail_activity`-trimmed for retention), the `StoreLock` (cross-process advisory lock, see Concurrency below), and the on-disk layout (`<root>/<boardId>/{board.json, lists.json, labels.json, cards/<id>.json, attachments/<cardId>/â€¦, activity.log}` + a top-level `<root>/.lock`)
  - `__init__.py` â€” `get_backend()` factory (cached singleton); selects `trello` (default) or `local` from `--backend` / `TRELLO_BACKEND` (see DESIGN.md)
- `trello_cli/fmt.py` â€” compact table/detail formatting and small helpers (`short_id`, `truncate`, `due_str`, `label_str`, `is_image`, `size_str`, `print_json`); backend-agnostic (formats plain dicts)
- `trello_cli/main.py` â€” CLI entry point, noun-group dispatch (`card`, `list`, `label`, `checklist`, `comment`, `attachment`), name/ID prefix resolution. **Board management** (`cmd_board`): besides `board`/`board add`, the verbs `board rename <name>`, `board archive` (soft delete), and `board restore`/`board unarchive` forward to `api.update_board` on the `--board` board; `boards --archived` lists only archived boards and `boards --all` lists both with a State column (default `boards` stays open-only). `_resolve_board_ref` now lists with `include_closed=True` so an archived board can still be addressed (e.g. to restore it). Top-level workflow command `grab` (`cmd_grab`): `trello grab [--from "To Do"] [--to "Doing"]` claims the top card of one list and moves it to another via `api.grab_top_card`, printing the card it got (or the full dict under `--json`) and exiting 1 when there's nothing to grab â€” the one-command form of CONTRIBUTING.md's claim handshake, for telling several agents to "grab the top ticket" without them colliding. The `export` command bridges two backends in one invocation â€” it reads a board snapshot via the `api` facade (shared `_gather_board` helper) and writes via a directly-instantiated *other* backend, bypassing the `get_backend()` singleton â€” and runs in two directions on `--to`:
  - `--to local` (default, `_export_to_local`): pull from the selected backend (must not be local) into a `LocalBackend`. Also downloads uploaded attachment blobs (default on; `--no-attachments` skips) via the *source* backend's authed `download_attachment`, rewriting each card's attachment url root-relative *before* `import_board` persists it â€” best-effort (per-blob failure warns + keeps the remote url; immutable blobs are reused on re-export). IDs preserved â†’ idempotent in-place refresh
  - `--to trello` (`_export_to_trello` â†’ `_push_board_to_trello`): push the local store (source backend must be `local`) **up to Trello as a brand-new board**. Trello mints fresh ids, so ids can't be preserved â€” it builds oldâ†’new maps for labels/lists, then re-creates each card and its children (comments, checklists+items, attachments) under the new ids via the ABC's write ops. **Create-new-each-time** (non-idempotent; no localâ†’trello map persisted â€” keeps statelessness). Re-uploads local blobs (`add_attachment_file`), re-links URL attachments; comments are re-posted as the token user with an `_originally <author>, <date>_` provenance prefix (Trello can't set author/date); `idMembers` dropped; only open lists pushed (`--name` overrides the new board name). A tracked-mapping re-sync (`--into <board>`) is **designed but not implemented** â€” DESIGN.md works through the id-map sidecar, the diff/reconcile algorithm, and conflict handling, and recommends deferring the build pending a user decision on the conflict policy (it can't be verified against live Trello here)
- `trello_cli/web/` â€” optional web app (the `[web]` extra: fastapi, uvicorn, watchdog, python-multipart â€” the last for browser file uploads), launched by `trello serve`:
  - `server.py` â€” FastAPI JSON API mapping 1:1 onto the `api` facade, so it renders **either** backend with no per-backend code. Endpoints: `GET /api/boards` (`?include_closed=true` adds archived boards for the manage-boards panel), `GET /api/boards/{id}` (board+lists+cards), `PATCH /api/boards/{id}` (rename `name` / archive-restore `closed`), `DELETE /api/boards/{id}?confirm=true` (permanent purge â€” `confirm=true` required, the web echo of `local rm --yes`; local-only via the facade guard), `POST /api/boards/{id}/lists` (add column), `GET /api/boards/{id}/labels` + `POST /api/boards/{id}/labels` (list/create board labels), `GET /api/cards/{id}` (detail+comments), `PATCH /api/cards/{id}` (drag move/reorder **and** detail-panel edits â€” title/desc/due/dueComplete), `DELETE /api/cards/{id}` (soft delete = archive), `POST /api/cards/{id}/comments` (add comment), `POST /api/cards/{id}/labels` + `DELETE /api/cards/{id}/labels/{labelId}` (add/remove a label, returns the fresh card), `POST /api/cards/{id}/attachments` (add a URL attachment) + `POST /api/cards/{id}/attachments/file` (multipart upload â€” saves to a temp file then `add_attachment_file`) + `GET /api/cards/{id}/attachments/{attId}/raw` (serve an **uploaded** blob's bytes â€” a local blob, or a Trello-hosted upload fetched with OAuth â€” streamed via `download_attachment` into a temp file and returned inline; external URL attachments are **not** proxied, the browser links them directly, so the server never fetches an arbitrary URL) + `DELETE /api/cards/{id}/attachments/{attId}` (remove); the add/delete routes all return the fresh card, `PATCH /api/lists/{id}` (column reorder `pos`, archive `closed`, persisted `sort`), `POST /api/lists/{id}/cards` (create), `GET /api/events?board=<id>` (SSE live-refresh stream; the `?board=` lets the Trello backend poll that board). ## Conventions

- **Noun-group dispatch** â€” `_dispatch(group, subcmds, args)` routes verbs within a group. Bare nouns (or nouns followed by a non-verb) fall back to `ls` if the group has one, so `trello list` â‰¡ `trello list ls`.
- **Resolvers** â€” every domain has a `_resolve_*` helper that accepts an ID, an ID prefix, or a case-insensitive name prefix, and raises `SystemExit` on miss/ambiguity.
- **Board scope** â€” `_require_board()` resolves the board from `--board <name_or_id>` (parsed in `main()`) or the `TRELLO_BOARD` env var, and errors if neither is set. There is no stored "active board" (see Statelessness).
- **Backend scope** â€” `get_backend_name()` selects the data source from `--backend <trello|local>` (parsed in `main()`) or the `TRELLO_BACKEND` env var, defaulting to `trello`. The local store folder resolves as `--local-root` flag > `TRELLO_LOCAL_ROOT` env > config `local_root` > `~/Dropbox/trello-cli`; `trello local init [path]` creates it and persists `local_root`.
- **Statelessness (design guideline)** â€” the CLI is used by many agents and projects concurrently, so it keeps **no shared mutable session state**. Selection (board, backend) is always per-invocation via flags/env; only stable config (credentials, `local_root`) is persisted. Don't add "active X" state â€” that creates cross-invocation conflicts. (The legacy active-board feature was removed for exactly this reason; everything uses `--board`.)
- **Output mode** â€” `--json` is stripped in `main()` and toggles `_JSON_MODE`; read commands branch on `_is_json()` to emit raw JSON via `print_json` instead of formatted tables.
- **Backend seam** â€” commands never touch a concrete backend or transport. They call `api.<op>(...)`, which forwards to `get_backend()`. Every backend returns the same Trello-shaped dicts (the keys `fmt.py` reads), so adding a backend means implementing the `Backend` ABC â€” no command or formatter changes.
- **Concurrency (local backend)** â€” many agents run the CLI at once, so every `LocalBackend` *write* is a read-modify-write that would otherwise lose updates (and crash on Windows' `os.replace`) when two processes race the same file. `StoreLock` (in `store.py`) serializes them: a cross-process OS advisory lock on `<root>/.lock` (`fcntl.flock` / `msvcrt.locking`, auto-released on process death) plus an in-process `threading.RLock` for re-entrancy + thread-safety, with a bounded blocking wait. It wraps the whole loadâ†’modifyâ†’save of each mutator (the `_MUTATORS` list). **Adding a new mutating op? Add its method name to `_MUTATORS`.** Reads are intentionally lock-free (atomic writes already give each file a consistent view). Scope: same-machine multi-process. Cross-machine Dropbox edits stay last-write-wins (OS locks don't cross machines) â€” see DESIGN.md.
